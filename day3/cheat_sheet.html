Three jobs the encoder cannot easily do

Job 01

Free-text extraction

Turn a paragraph into structured fields the encoder cannot list as labels in advance. Signatories, agreement type, year, sector.

Job 02

Normalization

Resolve "USA," "U.S.," "United States" to one canonical form across messy crawled text. One prompt with examples.

Job 03

Light summarization

Pick three or four facts out of a longer paragraph where the field set varies by document.

Validate

CSV per stage. Audit log. Raw HTML. Cost summary.

Schema design · decide before any code runs

{ "signatories": list[str], # normalized to short country names "agreement_type": enum, # bilateral_trade, climate, defense, ... "year": int | null, # 1990-2030 or null "sector": enum, # trade, climate, energy, ... }
Three properties. The schema is finite (decided in advance, adding a field means re-running every call), typed (every field has a type and, where possible, a closed taxonomy), and pinned (documented in the system prompt and in the replication package).

Where it lives. The system prompt holds the schema and one example of valid output. The example is the highest-leverage line. Cache the system prompt to bill at ~10% of base input rate.

Validation without a gold set · four checks for free

Check 01

Schema

Parses as JSON. All required fields. Types and taxonomies match. Programmatic.

Check 02

Determinism

Two runs at temperature 0. Count identical outputs. The drift is your floor.

Check 03

Sample audit

Read 5 random outputs by hand. Smallest defensible spot-check.

Check 04

Audit log

One row per call. Lead, prompt hash, response, tokens, cost.

Reproducibility appendix · what to ship with the paper

Must-have

Column 01

  • Full prompt verbatim, system + user, with date finalized.
  • Model ID with version pin. claude-haiku-4-5-20251001
  • Temperature, max_tokens, any other generation parameter set.
  • Schema with named taxonomies.
  • Schema-validation pass rate on the full corpus.
  • Robots.txt at date of crawl, plus your User-Agent string.
  • Raw-HTML files in package or stable archive mirror.
  • Audit log with one row per call.
Often asked for in revisions

Column 02

  • Hand-coded gold set with field-level F1 against the model.
  • Determinism rate from running the pipeline twice.
  • Subgroup pass-rate breakdown. Sector, region, length.
  • Cost summary at production scale.
  • Simpler-baseline comparison. Rule-based or smaller model.
  • Rerun against an updated model after publication.

Crawl etiquette · the 5 things in every fetch

User-Agent with a real contact email. Timeout of 10 seconds. Status check, 200 is the only happy path (403 blocked, 429 slow down, 503 try later). Raw HTML stored on disk after every fetch. Sleep at least one second between requests, slower under 429. Read robots.txt on screen and save a copy with the date. Cache by URL hash so re-runs do not re-fetch.

One full row · the unit of replication

url + fetched_at + status + raw_path + headline + date + n_paragraphs + extracted_json (4 fields) + model_id + system_prompt_hash + input_tokens + output_tokens + validators_passed + cost

A reviewer can take any one row and trace it back to a fetched HTML file, a parser version, a model version, an exact prompt, and a token cost. Twelve months from now, you will thank past-you who saved every column.

Pitfalls

  • Generative for closed-label tasks. If the answer fits a list, an encoder is cheaper, faster, and more deterministic.
  • No schema. Without a pinned schema, every call invents reasonable but inconsistent values. Closed taxonomies fix this.
  • No prompt caching. The system prompt is identical across calls. Caching bills it at 10% of base.
  • API key in code cells. Use userdata.get. Never paste a key into a notebook.
  • Crawl without robots.txt. Read it on screen. Save a copy. Document the date.
  • No raw-HTML storage. Re-running the parser on stored HTML beats re-crawling every time.
  • Skipping the audit log. Without per-call logging, the appendix has nothing to ship.
