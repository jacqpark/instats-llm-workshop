Day 305 / 30
The three jobs, with the diagnostic question

One question tells you
which family fits.

Diagnostic 01

Closed list?

Can the answer be expressed as a value from a finite list you wrote down before reading any text?

Yes. Encoder.
No. Generative.

Diagnostic 02

Messy input?

Is your input messy in a way an encoder hypothesis cannot fix? Country-name variants, date formats, alias resolution.

Yes. Generative.

Diagnostic 03

Variable schema?

Do you need three or four pieces of information that may or may not appear, with different surface forms each time?

Yes. Generative.

If two of three are yes, the generative tool is almost certainly the right starting point. If two of three are no, run an encoder first and reach for a generative model only if the encoder ceiling is hit.

Day 3 · Movement 0106 / 30
Three real costs that come with the new tool

Tradeoffs that go
in your replication appendix.

Cost 01

Per-token billing

Generative API calls are billed per input and output token. Today's demo on Anthropic Haiku 4.5 costs roughly five US cents for fifty leads plus five end-to-end articles.

Fine for a paper. Becomes a research-design constraint at million-paragraph corpus scale.

Cost 02

Non-determinism

Even at temperature zero, two calls with the same prompt can produce slightly different outputs. Providers also retire models behind the scenes.

Pin model version. Run the determinism cell. Report the number.

Cost 03

Prompt versioning

The prompt is a new artifact in your replication package. If the prompt changes, the output changes. Same for the model version.

Ship system prompt verbatim. Include date finalized. Hash it for the audit log.

None of the three costs existed in yesterday's encoder appendix. All three are now part of the methods burden.

Day 309 / 30
The one-hour decision that makes everything else defensible

Decide the schema
before you touch any code.

{ "signatories": list[str], # parties to the agreement "agreement_type": enum, # closed taxonomy, 12 values "year": int | null, # 1990-2030 or null "sector": enum, # closed taxonomy, 13 values }

Three properties

  • Finite. The field set is decided in advance. Adding a new field after the fact means re-running every call.
  • Typed. Every field has a type and, where possible, a closed taxonomy. The validator in section 5 of the notebook reads off the type signatures.
  • Pinned. The schema is documented in the system prompt and in the replication package. Future-you needs both.
Day 3 · Movement 0210 / 30
Two prompts, one extraction

System prompt holds the schema.
User prompt holds the lead.

System prompt · cached · same every call
You are an information extraction system. Extract a JSON object with exactly four fields. "signatories": list of strings, normalized to short country names. "agreement_type": one of [bilateral_trade, multilateral_trade, climate, defense, health, financial_assistance, IP, security_cooperation, infrastructure, technology, diplomatic_normalization, other]. "year": four-digit integer or null. "sector": one of [trade, climate, defense, health, finance, IP, security, infrastructure, technology, energy, agriculture, migration, other]. Return ONLY the JSON object. Do not wrap in markdown. Example input: "On 12 March 2024, France and Kenya signed a bilateral health-sector aid agreement..." Example output: {"signatories": ["France", "Kenya"], "agreement_type": "health", "year": 2024, "sector": "health"}
Cached at ~10% of base rate across 50 calls. The example is the highest-leverage line.
User prompt · per row · not cached
"On 8 September 2020, the United Arab Emirates and Israel signed the Abraham Accords, normalizing diplomatic relations and establishing cooperation in security, technology, and tourism."
One news lead per call. Variable. Not cacheable.

Day 3 · Movement 0212 / 30
Validation, no-hand-coding edition

Four checks you get for free
without writing any gold.

Check 01

Schema validation

Every output parses as JSON. Every required field is present. Every value matches its declared type or taxonomy. Programmatic.

Check 02

Determinism

Same prompt, two runs, temperature zero. Count identical outputs. The drift you see is the floor on reproducibility.

Check 03

Sample audit

Read five random outputs by hand. Ten minutes of human time. Smallest defensible spot-check.

Check 04

Audit log

One row per call. Lead, prompt hash, timestamp, response, token counts, cost. The replication artifact.

Necessary, but not sufficient. These four checks are prerequisites for any further validation. They do not replace a hand-coded gold set. Slide 28 is where the gold set comes back as the missing piece for a publishable paper.

# https://www.csis.org/robots.txt User-agent: * Allow: /analysis/ Allow: /*?page= Disallow: /admin/ Disallow: /search/ Disallow: /user/ Disallow: /comment/ User-agent: Bytespider Disallow: / # Verified 2026-04-28. Re-check # before any second-pass crawl.

Three things to find

  • User-agent. Which crawlers each rule block applies to.
  • Allow. Paths you may fetch.
  • Disallow. Paths you may not fetch.

Why it matters before any code

Robots.txt is a social contract. It does not have legal force in most jurisdictions. It is the first thing a methods reviewer will ask whether you checked. Document the date you read it. Save a copy. Re-check before any second pass.

Day 3 · Movement 0318 / 30
Five things in every fetch

The Python is short.
The hygiene is the whole job.

# A polite fetch in 8 lines. import requests, time UA = 'workshop-demo (mailto:you@univ.edu)' def polite_fetch(url): r = requests.get(url, headers={'User-Agent': UA}, timeout=10) if r.status_code != 200: return None time.sleep(1.0) return r.text # raw HTML
  • User-Agent. A real string that identifies you and gives a contact email. Sysadmins should be able to reach you instead of blocking your IP.
  • Status check. 200 is the only happy path. 403 means blocked. 404 means moved. 429 means slow down. 503 means try later.
  • Timeout. Default 10 seconds. Without it your script hangs on a slow page and never tells you why.
  • Raw HTML. Store the bytes, not just the parsed text. Slide 21 explains why.
  • Sleep. One second between requests is the floor. Slide 22 details politeness limits.

Day 3 · Movement 0320 / 30
The reproducibility move

Save the raw HTML
after every fetch.

Reason 01

Parsers change

You will discover next month that the date selector also matches a sidebar element. Re-running the parser on stored HTML is a one-line script. Re-crawling the site is a one-day script and a fresh robots check.

Reason 02

Sites change

The article you scraped today may not look the same in six months. Stored raw HTML is the only durable evidence that your data existed in the form you claim.

Reason 03

Replication packages

You ship the parser plus the raw HTML. A reviewer running your code in 2030 should not need network access to reproduce your dataset.

Filename is the URL slug plus a timestamp. Compress as gzip if the corpus grows.

Day 3 · Movement 0321 / 30
The simple rule

Be slower than you think
you need to be.

Rates of fire

PaceVerdict
0.5 / secPolite. Default for production crawls.
1 / secFloor. Today's notebook setting.
5 / secRude on most sites. Avoid.
20 / secDenial of service. Never.

Three implementation details

  • Sleep after each request rather than before the first. The point is to space the load on the server. Spacing your own script is a side effect.
  • Cache responses by URL hash. If you re-run, do not re-fetch what you already have.
  • Log every request with a timestamp. If a sysadmin emails you, you can show your traffic pattern from your own logs.

CSIS does not publish explicit rate limits. The default in the demo is one second between requests. Tune slower if your IP starts catching 429 responses. Never speed up.

The unit of replication

What gets saved.
Twelve columns. One row.

# One full row, all columns url: https://www.csis.org/... fetched_at: 2026-04-28T09:14:22Z status: 200 raw_path: raw_html/ai-and-future....html headline: "AI and the Future of Mediation" date_text: Apr 27, 2026 n_paragraphs: 14 extracted_json: {"signatories": ["UN", "WTO"], "agreement_type": "technology", "year": 2026, "sector": "technology"} model_id: "claude-haiku-4-5-20251001" system_prompt_hash: a1b2c3d4... input_tokens: 412 output_tokens: 48 validators_passed: True

Why this row

A reviewer can take this row and trace it back to a fetched HTML file, to a parser version, to a model version, to an exact prompt, to a token cost.

Twelve months from now

You will thank the past version of you who saved every column. The cost of saving more is zero. The cost of saving less is a request for the appendix you did not write.

Provenance for one row equals provenance for the dataset. The pipeline either saves it for every row or it saves nothing usable.

Day 327 / 30
For publication

The reproducibility appendix
for an LLM-based measurement paper.

Must-have

  • The full prompt verbatim, system and user, with date finalized.
  • Model ID with version pin. Today's pin is claude-haiku-4-5-20251001.
  • Temperature and any other generation parameters set.
  • Schema with named taxonomies.
  • Schema-validation pass rate on the full corpus.
  • Robots.txt at date of crawl, plus your User-Agent string.
  • Raw-HTML files in the replication package, or a stable archive mirror.
  • Audit log with one row per call.

Often asked for

  • A small hand-coded gold set with field-level F1 against the model.
  • A determinism rate from running the pipeline twice.
  • Subgroup pass-rate breakdown. Sector, region, length.
  • Total cost summary at production scale.
  • Simpler-baseline comparison. Rule-based or smaller model.

If column one is in the appendix, the pipeline is defensible. Column two is where revisions go.
